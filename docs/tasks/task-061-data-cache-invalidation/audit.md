#### Non-blocking observations (style/optional)
- The atlas-maps `SpawnPointRegistry.FlushTenant` (`map/monster/registry.go:293-333`) duplicates the SCAN+pipelined-DEL algorithm that now lives in `libs/atlas-redis/tenant_registry.go:189-223`. The duplication is justified — the spawn-point key shape `atlas:maps:spawn:{tenantId}:{worldId}:{channelId}:{mapId}:{instance}` is hand-rolled and does not flow through `tenantEntityKey`/`tenantScanPattern`, so `TenantRegistry.Clear` cannot be reused without first migrating the spawn-key shape. The design.md (§4.2) acknowledges this. Not a violation; flagged only so a future reader doesn't assume it can be dedup'd by deleting the wrapper.
- `dataUpdatedEventBody` (`services/atlas-data/atlas.com/data/data/kafka.go:26-30`) is duplicated in both consumer-side `kafka.go` files (`atlas-monsters/.../kafka.go:15-19`, `atlas-maps/.../kafka.go:15-19`). This mirrors the existing repo convention for Kafka envelopes — each service redeclares the contract types so changes break compilation on every consumer. Not a DOM-21 violation; just noted for completeness.
- `eventsConsumerErrorsTotal` and `eventsSkippedTotal` carry a free-form `tenant` label on `keysDeletedTotal` only (`atlas-monsters/.../metrics.go:24-27`, `atlas-maps/.../metrics.go:24-27`). Tenant cardinality is bounded in this deployment, so this is acceptable; mentioning it because high-cardinality Prometheus labels are usually flagged.
