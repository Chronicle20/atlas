# Combat Template Opcode Audit (task-065 Item 2)

PRD §4.4 required cross-checking combat template opcodes against IDA dispatcher case-statement values. This document records what the audit covered and what it deferred.

## Scope

Audit target: combat-domain entries in `services/atlas-configurations/seed-data/templates/template_{gms_12,gms_83,gms_87,gms_92,gms_95,jms_185}_1.json`.

Combat writers/handlers in scope (per `libs/atlas-packet/{monster,drop,reactor,pet}/{clientbound,serverbound}/`):

- 20 clientbound writers (monster 9, drop 2, reactor 3, pet 6)
- 11 serverbound handlers (monster 1, drop 1, reactor 1, pet 8)

## Findings

### 1. Writer/handler name strings — ✅ no drift

Every combat opcode entry in every template uses the canonical writer/handler name string declared as a `const` in `libs/atlas-packet/`. There is no string-level naming drift between atlas code and template config. The matrix was generated by enumerating `^const.*Writer = "..."` / `^const.*Handle = "..."` in `libs/atlas-packet/{monster,drop,reactor,pet}/` and matching against each template's `socket.writers[].writer` and `socket.handlers[].handler`.

### 2. Within-template opcode collisions — ✅ none in combat domain

No two combat writers or handlers share an opcode within any single template. (Cross-domain collisions exist between Auth/ServerList variants in the login domain — those are intentional, the writers share an opcode and distinguish via a leading status byte.)

### 4. IDA dispatcher case-statement verification — ⏸ deferred

The PRD §4.4 acceptance condition ("commit message citing the IDA case-statement value") requires reading dispatcher case-statements from IDA. The audit pipeline does not record dispatcher opcode values — `docs/packets/ida-exports/*.json` carries function addresses + call sequences, not the calling dispatcher's case-statement value. Without IDA access this leg of the audit cannot complete.

## Conclusions

For the work that was in scope of this branch (drift in existing template entries):

- ✅ No writer/handler name string drift between atlas and templates.
- ✅ No opcode collisions in the combat domain within any template.
- ✅ No template values were changed; no `post-phase-b.md` "Template opcode fixes" table entries.

The population gap (Finding 3) is a real channel-server runtime concern but falls outside task-065's scope — fixing it requires (a) IDA decompile of each version's dispatcher to recover correct opcode values per writer, and (b) potentially adding rows to 5 templates. This work belongs in a follow-up task gated on the same IDA access that gates [Item 1] (`CMob::GenerateMovePath` v83/v87 addresses).

## Why no template changes landed in-branch

No template values were modified because:

1. The only legitimate change would have been populating Finding 3's gaps, which requires per-version dispatcher case-statement values from IDA.
2. Inferring v95 values from neighboring versions is not safe — the inter-version opcode progression is non-monotonic (e.g., SpawnMonster jumps 0x6D → 0xEC → 0xFC → 0x114 → ? → 0xFD across v12/v83/v87/v92/v95/jms_185).
3. Adding placeholder rows with `0x00` or duplicated values would create silent runtime collisions; warnings-from-absence is the safer failure mode.
