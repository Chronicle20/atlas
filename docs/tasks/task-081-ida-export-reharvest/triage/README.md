# Task-081 ‚Äî four-version audit triage worklist

Generated by `packet-audit triage` (commit `c139d301a`), run live against all four IDB
instances (v83/v87/v95/jms185) over every NON-‚úÖ audited packet. The goal: turn ~330
non-‚úÖ audit rows into an actionable, conditional-aware backlog.

## How each packet is classified
For each non-‚úÖ packet, `triage` decompiles the client function (live) and:
- **`per-mode-branch`** ‚Äî the handler BRANCHES on an early read (`switch`, or a bare
  `if (byte == N)` like `CDropPool::OnDropLeaveField` / DropDestroy) or reads inside a
  loop. A FLAT read-order comparison is INVALID for these (the exporter flattens all
  branches; Atlas writes one). NOT a real bug ‚Äî needs per-branch verification via the
  dispatch-selector path. **The largest bucket ‚Äî these are why the raw ‚ùå count is
  misleading.**
- **`candidate-real-divergence`** ‚Äî a FLAT (non-branching) handler whose verified client
  read-order genuinely mismatches Atlas's writer (op/width/length) and is NOT width-
  tolerable. The actionable likely-bug list.
- **`representation`** ‚Äî flat + every mismatch is width-tolerable (same bytes, different
  decomposition). Accept.
- **`unverifiable`** ‚Äî decompile failed / `Unresolved`-dominated.

## IMPORTANT ‚Äî this is a STARTING list, not a confirmed bug list
The conditional heuristic is imperfect; the `candidate-real` bucket has KNOWN false
positives that still need IDA verification before fixing:
- **Loop-heavy handlers** mis-bucketed as flat (e.g. `CMobPool::OnMobEnterField` ‚Äî the
  repeated `Decode4,Decode2,Decode2` run is a movement-path loop; if the decompiler
  unrolled it, `ReadsAreConditional` doesn't see a loop construct). These are really
  loop/struct cases, not divergences.
- **`#`-mode entries** that slipped into `candidate-real` with an empty/odd read (e.g.
  `CUIGuildBBS::OnGuildBBSPacket#BBSThread`) ‚Äî a `#` entry's base IS a switch handler ‚Üí
  should be `per-mode-branch`. `triage` should special-case `#` FNames.
Refinements (loop detection on unrolled reads; `#`-aware classification) are registered
follow-ups. Until then: **triage each `candidate-real` entry in IDA** ‚Äî confirm the flat
client read-order, then decide real-Atlas-bug (fix `libs/atlas-packet/...` + byte test) /
mode-or-loop (false positive) / representation.

## Workflow per `candidate-real` entry
1. The entry gives the FName + address + the verified client read-order.
2. Decompile in IDA; confirm it's genuinely flat (no branch/loop on an early read).
3. Compare to the Atlas writer (the committed audit `.md`/`.json` row detail has
   atlas-op vs client-op per position).
4. If a real wire bug ‚Üí fix the encoder to match the client read + a per-version byte
   test (`pt.Variants`/`pt.RoundTrip`), re-audit ‚Üí ‚úÖ.
5. If a mode/loop/representation false positive ‚Üí note it (and feed back into the
   `triage` heuristic).
