# Task 107 — Door Packet-Fixture Campaign — Context

Companion to `plan.md`. Key files, decisions, and dependencies an executor needs.

## What this task is (and is not)

- **Is:** a packet *verification* campaign — promote 12 `door/clientbound` coverage-matrix cells (3 packets × gms_v84/v87/v95/jms_v185) from `incomplete` to `verified` (✅) by producing the evidence the grader consumes.
- **Is not:** a feature. No new door behavior, no encoder change expected. The encoders are already byte-correct; we are independently confirming each version's client truly reads the v83 layout and producing the matrix artifacts.

## Decisions locked in (from design)

1. **Writer ownership** is `libs/atlas-packet/door/clientbound/` (single layout, no version branch). Production usage: `services/atlas-channel/.../socket/writer/door.go`.
2. **`RemoveDoor` ≠ `RemoveTownDoor`** — two distinct opcodes (`REMOVE_DOOR` 5 bytes; `SPAWN_PORTAL` town-removal 8 bytes), distinct bodies. Already pinned by v83 fixtures.
3. **Tiers:** `SpawnDoor`/`RemoveDoor` = tier-0 (report + marker, **no** evidence pin); `RemoveTownDoor` = tier-1 (report + marker + evidence pin). Source of truth = `status.json` `tier1` flag (the PRD's `(T1)` annotation is wrong; `status.json` wins).
4. **No version is `n-a`** — all three ops are in every registry with real opcodes; all 12 cells are real gaps.
5. **Commit granularity:** one commit per `packet × version` cell, carrying its export splice + report (+ evidence) + marker + regenerated `STATUS.md`/`status.json`. Sequence so each regen reflects exactly the cells landed so far.
6. **Group by IDB** — finish all 3 cells of one version before moving to the next; never interleave two versions against the shared IDA instance.

## The dependency that reshapes the work (design §3 — CONFIRMED in planning)

The three receiver functions (`CTownPortalPool::OnTownPortalCreated`, `...OnTownPortalRemoved`, `CWvsContext::OnTownPortal`) exist **only** in `gms_v83.json` — absent from the v84/v87/v95/jms exports (verified by grep during planning). Consequences for the executor:

- **Report-gen will fail "not in export"** for every target cell until the receiver is spliced in. The splice (plan P4) is mandatory, not optional.
- **Never run a full `export`** over a committed export file — it drifts ~150 unrelated keys (playbook §10). Use the targeted harvest (`--prior-export "" --pending <one-fname-roster>`) to a temp file, then splice absent-only.
- **The receiver may be unnamed** in v84/v87/v95/jms IDBs. Per memory: v95 well-named; v87 had naming groundwork; v84 byte-identical to v83; jms needs the `*_U_DEVM` build. Where unnamed → name it via the dispatcher case (opcode table in plan) + v83-twin structure match. Naming is producible, not a blocker.

## IDA access

- Use `mcp__ida-pro__list_instances` + `select_instance(port)`; **confirm the loaded binary name** — ports drift (memory IDBs_v9: v84=13337, v87=13340, v95=13339, jms=13338; v83=13341). Never hardcode/trust a port without checking the binary.
- The `packet-audit export` command talks to IDA over HTTP (`--ida-url http://192.168.20.3:<port>/mcp --ida-port <port>`) — point it at the SAME instance you confirmed.
- The marker `ida=0x<addr>`, the report `Address`, and the spliced export entry's address must all agree (else `matrix --check` flags an orphan marker).

## Verification gates (CLAUDE.md "Build & Verification")

- `go test -race ./...`, `go vet ./...`, `go build ./...` clean in `libs/atlas-packet` (the only module touched).
- `tools/redis-key-guard.sh` clean from repo root (unaffected by door changes).
- **No `docker buildx bake`** — no `go.mod` is touched; only `*_test.go` + docs change (design §10).
- `matrix --check`/`fname-doc --check`/`operations --check` bar = **no new problems** (pre-existing backlog makes a clean exit 0 unavailable); zero door-related orphan/dangling/stale/drift lines and the conflict count must not increase vs the Task 1 baseline.

## Risks (design §12)

- Export splice corrupting a committed export → absent-only surgical splice; `git diff` the export and confirm it only adds keys.
- Unnamed receiver → name via byte-signature + v83-twin match (producible).
- jms SMC → use the `*_U_DEVM` build; door receivers are simple so decompile is expected to succeed.
- Genuine read-order divergence from v83 → STOP, wire-fix-first as a separate commit (design §6), then resume the cell.
- `COutPacket` delegate harvest artifact blocking report-gen → strip the one delegate ctor call from the spliced entry (playbook §10).

## Out of scope (surface in PR, do not implement)

`SpawnPortal` (live town portal, 12 bytes) shares the `SPAWN_PORTAL` opcode but has no `status.json` op row, so it is not a tracked matrix cell. It has v83 evidence + a v83 report already. Do not add markers/reports/evidence for it on the target versions; flag it in the PR description as an untracked-but-evidenced writer for a possible future task.
