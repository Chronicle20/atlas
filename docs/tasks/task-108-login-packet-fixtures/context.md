# Task 108 — Login Packet-Fixture Campaign — Context

Companion to `plan.md`. Key files, decisions, dependencies, and the planning-time verified facts an executor needs.

## What this task is (and is not)

- **Is:** a packet *verification* campaign — promote 20 `login/*` coverage-matrix cells (`docs/packets/audits/status.json`) from `incomplete` to `verified` (✅), or the one jms user-limit cell to an IDB-justified `n-a`, by producing the artifacts the grader consumes (per-version audit reports, `packet-audit:verify` markers, evidence records, export splices).
- **Is not:** a feature. No new login/auth/world-select/PIN/SPW behavior. Production handlers (`atlas-login`, `atlas-channel`) are unchanged **unless** a decompile proves a real wire delta in a `libs/atlas-packet/login/` codec (then fix-first, plan §D.4) — no delta is expected except possibly jms `ServerListEnd`.

## The single most important planning finding

The design (`design.md`) claims login is "mostly deterministic report-gen with a tiny IDA surface." **That is false** — verified by grepping the committed exports. The base function names are present, but the analyzer needs the **`#suffix` synthetic export entries**, and those (plus `ChangeStepImmediate`) are ABSENT in several target exports. See `plan.md` §A for the full evidenced list. **Trust `plan.md` §A/§C over `design.md` §4.1/§5/§14 where they differ.** All four non-v95 IDBs (v83, v84, v87, jms) are needed, each with at least one surgical export splice.

## Decisions locked in (from design + planning)

1. **Module touched:** `libs/atlas-packet` only. Verification: `go test -race ./...`, `go vet ./...`, `go build ./...` clean. **No `go.mod` change → no `docker buildx bake`** (CLAUDE.md gate is conditional).
2. **Report filename = bare struct name** (no `Login` prefix). Marker `packet=` keeps `login/<dir>/`.
3. **`ServerListRequest` grades on report + marker, NO evidence** (verified siblings v87/v95/jms prove it; it's a sub-struct row).
4. **Serverbound op rows + tier-1 need report + marker + evidence**; clientbound tier-0 need report + marker only (no evidence unless `matrix --check` demands it).
5. **`matrix --check` is the arbiter; the bar is "no new problems"** (pre-existing conflict backlog makes exit 0 unreachable) — capture the Task 0 baseline and compare.
6. **One commit per `packet × version` cell** carrying its coupled artifacts + regenerated `STATUS.md`/`status.json` (shared state — sequence so each regen reflects exactly the cells landed so far). v84-VAC ×3 and v87-VAC ×3 may each be one commit (shared send fn / shared splice).
7. **Real wire delta → fix-first** (own commit in `libs/atlas-packet/login/`, update cross-version test, surface in `audit.md` + PR), never silently re-pin a ❌. jms `ServerListEnd` is the prime suspect.
8. **jms `ServerStatusRequest` is the one genuine fork** (design §6): name+splice if present-but-unnamed in the `*_U_DEVM` IDB, else IDB-justified `n-a`. Verify by trying — don't infer absence from the export gap.

## IDA discipline (plan Stage 2)

- `select_instance` is shared global state — do ALL of one IDB's cells, then switch. `list_instances` and confirm by **binary NAME**, never trust the port number (memory: IDBs_v9 ports v83=13341, v84=13337, v87=13340, v95=13339, jms=13338 — but confirm).
- jms: use the clean `*_U_DEVM` build (the retail dump is SMC/undecompilable, playbook §10).
- Export splice is **surgical, absent-only**; never re-run a full `export` over a committed file (drifts ~150 unrelated keys). `git diff` the export after every splice — only the intended keys may change.
- Strip any `{op: Delegate, ref: COutPacket}` ctor artifact from a spliced send entry (playbook §10).

## Dependencies / ordering

- Task 0 (baseline) before everything.
- Stage 1 (Tasks 1–4) needs no IDA; can run first. Any non-clean Stage-1 verdict escalates into the matching IDB task in Stage 2.
- Stage 2 grouped by IDB (Tasks 5 v83, 6 v84, 7 v87, 8 jms) — never interleave two IDBs.
- Task 9 (final gate) last; then `superpowers:requesting-code-review` before any PR (CLAUDE.md).

## Open risk

The biggest schedule risk vs the design's framing is that **this is a substantially larger IDA campaign than design.md implies** (4 IDBs, ≥5 splices), not a near-pure report-gen sweep. The two-stage structure still holds; the executor should expect real decompile/harvest work in Tasks 5–8, not just file copies.
