# Task 109 — Character Packet-Fixture Campaign — Context

Companion to `plan.md`. Key files, decisions, dependencies, and the planning-time verified facts an executor needs.

## What this task is (and is not)

- **Is:** a packet *verification* campaign — promote **47 `character/*` coverage-matrix cells** (`docs/packets/audits/status.json`) from `incomplete` to `verified` (✅), or to an IDB-justified `n-a` only where a function is genuinely version-absent, by producing the artifacts the grader consumes (per-version audit reports, `packet-audit:verify` markers, evidence records, export splices, and where needed a seed-template route or a codec fix).
- **Is not:** a feature. No new character lifecycle / spawn / movement / buff / keymap behavior. Production code (`atlas-login`, `atlas-channel`) is unchanged **unless** a decompile proves a real wire delta in a `libs/atlas-packet/character/` codec (then fix-first, plan §D.4) — jms `CharacterMovement` and jms `CharacterList` are the suspects; no GMS delta expected (v84 body ≡ v83 below ~0x3D).

## The single most important planning finding

Like the login sibling (task-108), **the design's optimistic classification is partly wrong and `plan.md` §A overrides it with source-verified facts.** Specifically: report filenames are the descriptive `name` from `run.go` (NOT `Character`-prefixed); many cells already have markers (the gap is report/fixture); status.json notes are imprecise (a "no audit report" cell can have a FlatInvalid report); and **three jms cells the design called Class-A are actually export-ABSENT** (`OnViewAllCharResult#suffix`, `SendStatChangeRequest`, `OnSetActivePortableChair`) needing jms-IDB splices. **Trust `plan.md` §A/§C over `design.md` where they differ.** All five IDBs are needed.

## Decisions locked in (from design + planning)

1. **Module touched:** `libs/atlas-packet` only. Gate: `go test -race ./...`, `go vet ./...`, `go build ./...` clean + `tools/redis-key-guard.sh`. **No `go.mod` change → no `docker buildx bake`** (CLAUDE.md gate is conditional).
2. **Report filename / marker last-segment = descriptive `name`** (no `Character` prefix for pkg-empty ops). Marker `packet=` keeps `character/<dir>/`.
3. **Every in-scope clientbound row is tier-1** (design §3) → report + marker + evidence. Serverbound → report + marker + evidence + routing. KeyMapChange sub-struct (v84, op=None) → mirror the op-row treatment, read `matrix --check`.
4. **`matrix --check` is the arbiter; bar is "no new problems"** (pre-existing conflict backlog makes exit 0 unreachable) — capture the Task 0 baseline and compare. Routing confirmed via `routedElsewhere && !routed` conflict, not by pre-emptive template edits.
5. **Phase A (Class C) = three NEW cross-version byte-fixture files** (`list/appearance_update/effect_quest_test.go`), full-body (nested avatar/look / effect mode body) — a length/mode-only assertion is a false pass (memory `feedback_dispatcher_mode_byte_is_false_pass`). File created in the v83 IDB task, appended per version (plan §D.7).
6. **One commit per `packet × version` cell** carrying coupled artifacts + regenerated `STATUS.md`/`status.json` (shared state — sequence so each regen reflects only landed cells). EffectQuest's two ops, AutoDistributeAp's two ops, and KeyMapChange op-vs-sub-struct each get their own fixture case / marker / evidence / commit.
7. **Real wire delta → fix-first** (own commit in `libs/atlas-packet/character/`, update cross-version test, surface in `audit.md` + PR), never silently re-pin a ❌. jms Movement / jms List are the suspects.
8. **Class E = name+splice, not n-a** (design §5): present-but-unnamed in the older/jms IDB (byte-signature + v87/v95 twin). `n-a` only if an IDB genuinely lacks the function after trying. Cells: CheckName v83/v84, Expression v83/v84, ChairShow v83/v84/jms, HealOverTime jms, ViewAll jms.

## IDA discipline (plan Stage 2)

- `select_instance` is shared global state — do ALL of one IDB's cells, then switch. `list_instances` and confirm by **binary NAME**, never trust the port (memory IDBs_v9: v83=13341, v84=13337, v87=13340, v95=13339, jms=13338 — but confirm).
- jms: the clean `*_U_DEVM` build (the retail dump is SMC/undecompilable, playbook §10).
- Export splice is **surgical, absent-only**; never re-run a full `export` over a committed file (drifts ~150 unrelated keys). `git diff` the export after every splice — only the intended keys may change. Strip any `{op: Delegate, ref: COutPacket}` ctor artifact from a spliced send entry.
- Distrust IDB symbol names for senders — ground truth is the `COutPacket::COutPacket(&pkt, OPCODE)` integer (playbook §10); relevant for HealOverTime jms (`SendStatChangeRequest` vs `...ByItemOption`).

## Open risk

The biggest risk vs the design's framing is the same as login's: **this is a larger live-IDA campaign than design.md implies** — all 5 IDBs, 20 fresh Phase-A decompiles, a Class-E cluster, and 5 jms/older splices, not a near-pure report-gen sweep. The phased structure holds; the executor should expect real decompile/harvest work in Tasks 4–8. The full-body Phase-A fixtures (CharacterList's nested avatar/look block especially) are the highest-effort items.
