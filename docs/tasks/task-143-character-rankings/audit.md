## Important

### DOM-27 — Transient DB errors map to 503, never bare 500 (atlas-rankings)
**FAIL.** `services/atlas-rankings/atlas.com/rankings/ranking/resource.go:72,79,99,106` correctly route every non-400/404 error through `server.WriteErrorResponse(d.Logger())(w)(err)`, which is the correct handler-side half of the pattern. But `services/atlas-rankings/atlas.com/rankings/main.go` never calls `server.RegisterTransientErrorClassifier(...)` — confirmed by `grep -rn "RegisterTransientErrorClassifier\|IsTransientConnectionError\|CountTransient" *.go ranking/*.go` returning zero matches in the whole module. Per `libs/atlas-rest/server/error.go:30`, "with no classifier registered, every error maps to 500" — so a transient Postgres pool-exhaustion error during `Recompute`/read paths surfaces as a bare 500, exactly the failure mode DOM-27 exists to prevent. Fix: add the three-line classifier registration to `main.go` alongside `database.Connect(...)`.

### DOM-28 — No silent degradation in decorators/enrichment (atlas-login)
**FAIL.** `services/atlas-login/atlas.com/login/character/processor.go:102-116` (`decorateRankings`) is a fallible remote-enrichment path (bulk rankings fetch) that, on error, logs a `Warn` and silently drops the fetched data — but never calls `degrade.Observe(...)`, so `atlas_enrichment_degraded_total` is never incremented for a ranking-fetch failure. This is the exact anti-pattern DOM-28 names ("A bare `if err != nil { return m }` ... is a finding regardless of justification"), made unambiguous by the sibling `InventoryDecorator` two functions below in the **same file** doing it correctly: `character/processor.go:169-182` wraps the equivalent inventory enrichment in `model.ErrDecorator(...)` with a `degrade.Observe(p.l, "login.character.inventory", m.Id(), err)` callback. `decorateRankings` is not even implemented as a `model.Decorator[Model]` (it's a private helper invoked inline from `GetForWorld` at line 93), so it bypasses the established decorator/metric pipeline entirely rather than being an exception to it. Fix: route the rankings enrichment through `model.ErrDecorator` (or add an explicit `degrade.Observe(p.l, "login.character.rankings", ...)` call in the existing warn branch) so a rankings-service outage is observable the same way an inventory-service outage is.

### DOM-05 — TransformSlice function exists, list handlers use it (atlas-rankings)
**FAIL.** `services/atlas-rankings/atlas.com/rankings/ranking/rest.go` defines `Transform(Model) (RestModel, error)` (lines 37-47) but no `TransformSlice`. Confirmed via `grep -rn "TransformSlice" .` across the whole module — zero matches. The bulk-list handler `handleGetRankingsForCharacters` in `ranking/resource.go:76` instead calls `model.SliceMap(Transform)(model.FixedProvider(ms))(model.ParallelMap())()` inline. This is exactly the composition the guideline treats as a substitute for, not an implementation of, `TransformSlice` — `rest.go` "Provide `TransformSlice([]Model) ([]RestModel, error)` for bulk transformations" (file-responsibilities.md:109) is unmet. Fix: add `TransformSlice` to `rest.go` and have `resource.go` call it.

### EXT-01 — Missing relationship-interface stubs on `atlas-rankings/tenant` REST client
**FAIL.** `services/atlas-rankings/atlas.com/rankings/tenant/rest.go` (the atlas-tenants REST client used by `tasks.RecomputeTask` to enumerate tenants) has no `SetToOneReferenceID`/`SetToManyReferenceIDs`/`SetReferencedStructs` methods — confirmed via `grep -n "SetToOneReferenceID\|SetToManyReferenceIDs" tenant/*.go` returning nothing, versus `character/rest.go:42-48` and `configuration/rest.go:32-42` in the *same service*, which both implement the stub trio (the latter's own comment at `configuration/rest.go:27-31` explicitly says it exists "to keep this client consistent... matching character.RestModel"). `libs/atlas-rest/CLAUDE.md` states these stubs are required "even when the caller doesn't care about the relationship payload," and that api2go's `jsonapi.Unmarshal` errors out (surfacing as a misleading generic failure) the moment the upstream response gains a `relationships` block. atlas-tenants' own `tenants` resource (`services/atlas-tenants/atlas.com/tenants/tenant/rest.go`) currently emits no relationships, so this is latent rather than live — but it is the same trap DOM-25/EXT-01's history (task-037, "GuildBBS", MTS msgType) explicitly warns is easy to reintroduce, and the sibling packages in this very PR already did the defensive thing. Fix: add the stub trio to `tenant/rest.go`.

### EXT-02 — No httptest-backed integration test for two of the three new REST clients
**FAIL (2 of 3 external clients).**
- `services/atlas-rankings/atlas.com/rankings/tenant/` has no `*_test.go` file at all (`ls tenant/*_test.go` → no matches). `character/` (same service) and `configuration/` (same service) both have httptest-backed tests (`character/processor_test.go:56-174`, `configuration/requests_test.go:27+`), so the tenant client is the outlier.
- `services/atlas-login/atlas.com/login/ranking/` (the new atlas-rankings client) has no `*_test.go` file at all (`ls ranking/*_test.go` → no matches). The unmarshal path (`ranking.Extract`, JSON:API decode via `jsonapi.Unmarshal` inside `requests.SliceProvider`) is exercised only indirectly, through `character/processor_rank_test.go`'s `rankingModel()` helper, which calls `ranking.Extract` directly on a hand-built `RestModel` — this never proves the wire format actually decodes via api2go, which is the specific gap EXT-02 and `libs/atlas-rest/CLAUDE.md` call out ("`FakeClient` mocks alone do NOT satisfy this — they bypass unmarshal"). Fix: add an `httptest.NewServer`-backed test to both packages serving a representative JSON:API fixture (matching `services/atlas-rankings/atlas.com/rankings/ranking/rest.go`'s actual wire shape for the login case) and asserting a populated result.

In practice `Build()` here is only ever fed from `Make(Entity)` (`entity.go:64-75`) after DB round-trip or from test fixtures, so the practical blast radius is low, but the documented contract is unmet — `TestBuilderRoundTrip` (`ranking/model_test.go:10-32`) only exercises the happy path, with no invalid-input case. Low severity because there is no demonstrated invariant an unvalidated `Build()` allows to reach persisted state.

Neither wraps its loop body in `t.Run(...)`, though (`for _, c := range cases { if got := ...; got != c.want { t.Errorf(...) } }` — no `t.Run`), so a failing case reports as one anonymous failure per test function rather than a named subtest. `testing-guide.md` does not hard-require `t.Run`; noting as a stylistic gap only.
